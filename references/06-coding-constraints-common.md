# 通用编码约束库（与语言无关）

> 本文件列出**与编程语言无关**的通用编码约束。Phase 1 生成规格书的 §6 时，按方案实际涉及的范围**摘取**条款落地，每条都必须可机械验证。

## A. 类型与单位

- **A1** 金额字段统一使用整型，单位为"分"，禁止使用浮点数。
- **A2** 时间字段统一使用 UTC，DB 列类型精确到毫秒；展示层负责时区转换。
- **A3** ID 字段统一使用 64 位整型；外部展示如需字符串（避免 JS 精度丢失）必须显式转字符串。
- **A4** 枚举值统一以"显式常量 + 显式数值"定义，禁止使用魔法数字。

## B. 超时与重试

- **B1** 所有数据库操作必须设置超时，默认 3 秒。
- **B2** 所有 RPC 调用必须设置超时，默认 1 秒（依方案具体规定为准）。
- **B3** 所有 HTTP/RPC 调用必须有重试上限，最多 2 次重试，退避策略明确（如 100ms / 200ms）。
- **B4** 仅幂等的下游调用允许重试；非幂等下游禁止重试，必须显式标注。
- **B5** 所有外部 I/O 必须传递 `context`（或等价 cancellation 机制）。

## C. 幂等与并发

- **C1** 所有写操作必须幂等。
- **C2** 幂等键的生成、存储、TTL、并发处理在规格书中显式规定。
- **C3** 乐观锁使用版本号字段，更新时 `WHERE version=? SET version=version+1`，并校验影响行数。
- **C4** 分布式锁仅在显式声明的临界区使用；持锁时间必须有上限，禁止"永不释放"。
- **C5** 禁止在 for 循环内做数据库/RPC 调用，必须使用批量接口或显式声明 N+1 风险已评估。

## D. 错误处理

- **D1** 错误处理保留完整错误链（语言相应的 wrap 机制）。
- **D2** 禁止吞掉错误（仅在显式标注"可忽略"的场景下允许，并必须打日志）。
- **D3** 每种业务错误对应一个明确的错误码 + HTTP 状态码 + 处理建议。
- **D4** 系统级错误（DB 连接断开、超时）与业务错误（库存不足）必须区分类型。

## E. 日志

- **E1** 日志必须为结构化（JSON 或等价键值）。
- **E2** 日志必须包含 trace_id（或等价的请求级唯一 ID）。
- **E3** 关键操作必须打日志（写库、写缓存、外部调用、主流程入口出口）。
- **E4** 敏感字段（密码、token、身份证、手机号、银行卡）必须脱敏。
- **E5** 日志级别使用规范：DEBUG 调试 / INFO 关键事件 / WARN 异常但可恢复 / ERROR 需告警。

## F. 配置

- **F1** 所有可变值（连接串、超时、限流阈值、特性开关）从配置文件/环境变量读取，禁止硬编码。
- **F2** 配置必须有默认值或启动校验，避免运行时才发现配置缺失。
- **F3** 敏感配置（密码、密钥）从安全存储读取，禁止明文写入仓库。

## G. 安全

- **G1** 所有外部输入必须做参数校验（类型、长度、范围、白名单）。
- **G2** SQL 必须参数化，禁止字符串拼接。
- **G3** 鉴权信息从安全的请求头/会话获取，禁止从查询参数获取敏感身份信息。
- **G4** 敏感数据传输使用 TLS。
- **G5** 错误信息对外暴露时必须脱敏（不暴露内部堆栈、SQL、文件路径）。

## H. 性能

- **H1** 所有外部调用都有明确的超时（参见 B 组）。
- **H2** 高频读路径优先走缓存，缓存策略（TTL、击穿、雪崩）在规格书中显式规定。
- **H3** 大批量数据接口必须支持分页或游标，禁止无界查询。
- **H4** 关键链路必须有可观测性指标（QPS、Latency、Error Rate）。

## I. 测试

- **I1** 每个产出代码模块必须有对应单元测试。
- **I2** 测试覆盖：正常路径 + 所有异常分支 + 边界条件。
- **I3** 外部依赖使用 mock 替代；mock 必须 mock 在接口层（不是具体实现）。
- **I4** 测试数据使用具有意义的命名（不要 `test123` `aaa`）。
- **I5** 测试必须可重复运行（无环境依赖、无运行顺序依赖、无外部网络依赖）。
