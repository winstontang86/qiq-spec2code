# Verifier 校验 8 大维度逐项 Checklist

> 本文件供 [@04-task-verification.md](04-task-verification.md) 使用。Verifier 必须把以下 8 个维度走完，不允许提前终止。

## 维度 1：结构符合性

### 1.1 文件结构
- [ ] 产出文件清单与 `TASK_CONTEXT.md` 中 `outputs` 完全一致（不多不少）。
- [ ] 文件路径与项目结构定义一致（包括子目录层级）。
- [ ] 文件命名符合规格书命名规范（如 `xxx_repository.go`）。

### 1.2 包结构
- [ ] `package` 名称与目录名一致。
- [ ] `import` 无未使用项（`goimports` 通过）。
- [ ] 无循环依赖。
- [ ] 不违反层次依赖规则：`domain` 不 import `infrastructure`、`application` 不 import `interfaces`。

## 维度 2：数据结构符合性

### 2.1 实体/结构体（**逐字段比对**）
- [ ] 结构体名称与规格书一致。
- [ ] 字段名称与规格书一致（含大小写）。
- [ ] 字段类型与规格书一致（注意：金额必须 `int64`、ID 必须 `int64`、时间必须 `time.Time`）。
- [ ] 字段 tag 正确：`json:"..."`、`gorm:"..."`、`db:"..."`、`validate:"..."`。
- [ ] 没有规格书中定义但代码遗漏的字段。
- [ ] 没有代码中存在但规格书未定义的字段。

### 2.2 枚举/常量
- [ ] 枚举名与值都与规格书一致。
- [ ] 没有遗漏值。
- [ ] 没有多余值。

### 2.3 状态机
- [ ] 所有合法转换都已实现。
- [ ] 所有非法转换被拒绝并返回错误。
- [ ] 转换前置条件检查完整。

## 维度 3：接口符合性

### 3.1 接口定义
- [ ] 方法名称与规格书一致。
- [ ] 参数列表（名称、类型、顺序）与规格书一致。
- [ ] 返回值（类型、顺序）与规格书一致。
- [ ] 第一个参数为 `ctx context.Context`。
- [ ] 最后一个返回值为 `error`。
- [ ] 没有遗漏或多余的接口方法。

### 3.2 HTTP 接口（如适用）
- [ ] 路径与规格书一致。
- [ ] HTTP 方法与规格书一致。
- [ ] 请求字段（名称、类型、必填、校验规则）与规格书一致。
- [ ] 响应字段与规格书一致。
- [ ] 错误码值、HTTP 状态码、message 与规格书一致。
- [ ] 规格书定义的所有错误码都已实现。

## 维度 4：流程符合性（**最重要**）

### 4.1 步骤完整性
- [ ] 规格书伪代码中的每一步都有对应代码（**逐步骤标行号**）。
- [ ] 步骤执行顺序与规格书一致。
- [ ] 没有规格书外的额外步骤（如有，必须合理且记录）。

### 4.2 异常处理
- [ ] 规格书中每个异常分支都有处理。
- [ ] 异常处理方式与规格书一致（如"库存扣减成功但订单创建失败时回滚库存"）。
- [ ] 没有未处理的异常路径。

### 4.3 事务边界
- [ ] 事务范围与规格书一致。
- [ ] 事务内操作清单与规格书一致。
- [ ] 事务失败处理与规格书一致。
- [ ] 事务外的回滚补偿（如反向 RPC 调用）已实现。

### 4.4 幂等性
- [ ] 规格书要求幂等的操作均已实现幂等。
- [ ] 幂等键来源、存储、TTL 与规格书一致。
- [ ] 并发同 key 请求的行为与规格书一致。

## 维度 5：编码约束符合性

> 标准来自 `TASK_CONTEXT.md` §6（精简的核心约束 + 沿用 `REPO_PROFILE.md` 风格基线的引用项）。日志、HTTP、mock 等风格类项不在此维度判定，统一对照风格基线。

### 5.1 核心约束（硬性）

- [ ] 金额字段使用 `int64`，单位"分"。
- [ ] 时间使用 UTC，DB 列精确到毫秒。
- [ ] ID 使用 `int64`。
- [ ] 枚举使用 typed const，无裸数字。
- [ ] 数据库操作设置超时（默认 3 秒，规格书可覆盖）。
- [ ] RPC/HTTP 调用设置超时（默认 1 秒，规格书可覆盖）。
- [ ] 仅幂等下游允许重试。
- [ ] 写操作幂等；幂等键来源/存储/TTL 与规格书一致。
- [ ] 乐观锁使用 version 字段，更新时校验 `RowsAffected`。
- [ ] 错误使用 `%w` 包装；不吞错。
- [ ] for 循环内无单条 DB/RPC 调用。
- [ ] 所有 I/O 传递 `context.Context`，生产路径无 `context.Background()` / `context.TODO()`。
- [ ] SQL 参数化，无字符串拼接。

### 5.2 nil 安全（必查）

- [ ] 返回指针/接口的函数，调用方均已显式判 nil 后再解引用。
- [ ] map 写入前已确认非 nil。
- [ ] 类型断言均使用 `v, ok := x.(T)` + ok 判定形式。
- [ ] JSON/RPC 反序列化的指针字段使用前已判 nil。
- [ ] **本任务相关代码 `nilaway` 单文件检查无报告**（如本地有 nilaway 工具）。

### 5.3 风格沿用（对照 `REPO_PROFILE.md` 风格基线）

- [ ] 日志库、字段命名、级别与基线一致。
- [ ] HTTP 框架、响应包装格式与基线一致。
- [ ] 测试 mock/断言库与基线一致。
- [ ] 错误返回风格与基线一致。
- [ ] 包名/文件命名与基线一致。

> 风格不一致默认为 Major 而非 Block；如基线本身缺失则降级为 Info 并记录到报告"基线建议"小节。

## 维度 6：测试覆盖性

- [ ] 存在 `xxx_test.go` 文件。
- [ ] 验收标准每条都有对应测试。
- [ ] 正常路径有测试。
- [ ] 异常路径有测试（每个异常分支至少 1 条用例）。
- [ ] 边界条件有测试（空、零、最大、最小、无效输入）。
- [ ] mock 正确：mock 在接口层、不 mock 具体实现。
- [ ] 测试数据具有意义，不是 `test123`。
- [ ] 测试可独立运行，无环境/顺序依赖。

## 维度 7：安全性基础

- [ ] 无 SQL 字符串拼接（参数化 SQL）。
- [ ] 敏感字段（密码、token、手机号、身份证）日志中已脱敏。
- [ ] 无硬编码密码/密钥。
- [ ] 用户输入有参数校验。
- [ ] 错误消息对外不暴露内部细节（堆栈、SQL）。

## 维度 8：反向检查（代码 → 规格书）

- [ ] 代码中无规格书未定义的功能/逻辑（若有，标注"额外实现"并评估）。
- [ ] 代码中无规格书未定义的字段/参数。
- [ ] 代码中无规格书未定义的错误码。
- [ ] 代码中无与规格书矛盾的实现。
- [ ] 所有 `// SPEC_QUESTION:` 注释已被记录到 `IMPL_REPORT.md` 偏差章节。

## 严重程度判定参考

| 维度 | Block 触发场景 | Major 触发场景 | Minor 触发场景 |
|---|---|---|---|
| 1 结构 | 文件路径错、产出缺失 | import 不规范 | 文件命名风格小差异 |
| 2 数据结构 | 字段类型错（如金额用 float64）、字段缺失 | tag 错误、字段顺序混乱 | 注释缺失 |
| 3 接口 | 方法签名错、错误码值错、HTTP 路径错 | 校验规则缺失、错误码 message 不一致 | 大小写差异 |
| 4 流程 | 关键步骤遗漏、事务边界错、幂等丢失 | 异常分支处理与规格书不一致 | 注释引用不准 |
| 5 约束 | 未传 ctx、未设超时、SQL 拼接、**nilaway 报告/裸类型断言/未判 nil 解引用** | 未用乐观锁、未幂等、风格基线不一致 | 注释引用不准 |
| 6 测试 | 无测试文件 | 异常分支无测试 | 测试命名不规范 |
| 7 安全 | SQL 拼接、敏感字段未脱敏、硬编码密钥 | 错误消息泄露内部信息 | — |
| 8 反向 | 与规格书矛盾的实现 | 规格书外的额外功能 | 规格书外的小工具函数 |
